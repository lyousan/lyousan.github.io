总览

上一篇我们讲到JAVA集合框架中的List接口，及其两个实现类ArrayList，LinkedList，也了解了这两个集合类的优缺点和适用场景。在集合框架中还有几个像List这样的父接口，今天我们要来分析的就是Set接口和Map接口。

前文中也提到了”不同的集合有着不同的特性，有的集合允许重复的元素，而有的不允许，有的集合是有序的，而有的是无序的“，那么我们就先从Set和Map的特性开始分析。

HashSet

This class implements the Set interface, backed by a hash table (actually a HashMap instance). It makes no guarantees as to the iteration order of the set; in particular, it does not guarantee that the order will remain constant over time. This class permits the null element.

根据官方的注释，我们可以了解到HashSet通过hash table实现了Set接口，而这个哈希表的具体实现其实就是HashMap，所以在这一小节中的部分知识会着重放在后文HashMap中。我们还可以了解到它不保证元素的顺序，特别是不保证元素的顺序会随着时间的推移保持不变，同时它允许null元素的存在。

可以看到HashSet这个类中的方法并不是很多，可以看到除了构造方法外，其他的方法都是继承自父类或实现于接口，它并没有向外扩展它自身的一些方法，接下来我们就先从HashSet的成员变量开始。

疑问

前面我们有提到关于Set集合的特点是无序且唯一，可是在HashSet的源码中我们却并没有看到任何与之有关的代码，大部分的方法都是草草地调用了HashMap中的方法，那么无序且唯一的特性又是怎么实现的呢？

HashMap

Hash table based implementation of the Map interface. This implementation provides all of the optional map operations, and permits null values and the null key. (The HashMap class is roughly equivalent to Hashtable, except that it is unsynchronized and permits nulls.) This class makes o guarantees as to the order of the map; in particular, it does not guarantee that the order will remain constant over time.

官方的注释直接翻译过来就是说：HashMap是基于（hash table）哈希表的实现的 Map 接口。 此实现提供了所有可选的Map操作，并允许 null的 值和 null键。 （ HashMap 类大致相当于 Hashtable ，除了它是不同步的，并允许null）。这个类不能保证Map的顺序; 特别是，它不能保证元素的顺序在一段时间内保持不变。但最后这句话我个人觉得其实应该理解为随着元素的数量变化它不能保证元素的顺序。

哈希表

关于HashSet和HashMap的官方注释中都提到了hash table也就是哈希表，也叫散列表，那这到底是个什么东西呢，我们就先从数据结构的层面来了解一下这个哈希表。

首先我们再来温习一下数组和链表的数据结构

  • 数组：查询和修改效率高，添加和删除效率低（根据具体情况而定）

  • 链表：添加和删除效率高，查询和修改效率低（根据具体情况而定）

我们再来看看哈希表长什么样？

一目了然，哈希表就是结合了数组和链表的一种数据结构，综合数组和链表的优点，所以查询、修改、添加、删除的效率都还可以。

而HashMap正是基于哈希表的一种实现，在jdk1.8之前HashMap是由数组加链表实现的，正好对应上面的哈希表的图，但是当链表上的元素越来越多时，查询的效率就变低了，于是在jdk1.8中HashMap在数组和链表的基础上加入红黑树的结构，以此来优化查询效率。

关于hashCode()这里做一个扩展

在StackOverFlow上有一个关于hashCode方法中乘数为什么是31的一个讨论，Why does Java’s hashCode() in String use 31 as a multiplier?

Ask Question

其中最高赞的回答是来自《Effective Java》这本书中的内容：

The value 31 was chosen because it is an odd prime. If it were even and the multiplication overflowed, information would be lost, as multiplication by 2 is equivalent to shifting. The advantage of using a prime is less clear, but it is traditional. A nice property of 31 is that the multiplication can be replaced by a shift and a subtraction for better performance: 31 * i == (i << 5) - i. Modern VMs do this sort of optimization automatically.

(from Chapter 3, Item 9: Always override hashcode when you override equals, page 48)

选择31是因为它是一个奇数素数。如果它是偶数，会出现乘法溢出，信息就会丢失，因为乘以2等于移位。使用素数的好处不太明显，但它是传统的。31的一个很好的特性是，可以用移位和减法替换乘法，以获得更好的性能:31 * i == (i << 5) - i。现代虚拟机能够自动执行这种优化。

还有一条引用了《Data Structures and Algorithms in Java》（数据结构与算法JAVA版）中的内容：

Goodrich and Tamassia computed from over 50,000 English words (formed as the union of the word lists provided in two variants of Unix) that using the constants 31, 33, 37, 39, and 41 will produce fewer than 7 collisions in each case. This may be the reason that so many Java implementations choose such constants.

See section 9.2 Hash Tables (page 522) of [Data Structures and Algorithms in Java](https://enos.itcollege.ee/~jpoial/algorithms/GT/Data Structures and Algorithms in Java Fourth Edition.pdf).

经过大量的实验计算，发现使用常数31,33,37,39和41将在每种情况下产生少于7个碰撞，这可能是许多Java实现选择这样的常量的原因。

对比了小傅哥关于hashCode做的相关测试后，使用31做乘数在不造成数据丢失的前提下的确是散列效果最好的常数。

也就是说选用31作为乘数的原因主要有三点：

  • 31是一个奇数素数，如果是偶数可能会导致信息丢失。
  • 31可以使用位运算和减法替换乘法，能够更好的性能。
  • 31做乘数能够在不造成数据丢失的情况下达到最好的散列效果。

总结

分析完HashMap的相关源码后，对于前文中提出的疑问：“HashSet是如何实现元素无序且唯一的？”，现在就很好理解了。HashSet正是通过HashMap来实现的，在HashSet中，我们存放的元素其实是放在HashMap里的，我们存放的元素就是HashMap中的Key，而Value就是用一个Object对象做占位。

  • 无序：HashMap中元素的添加是根据hash算法来确定元素要存放的位置，因此并不保证元素在集合中顺序与我们添加的顺序一致，而且随着元素数量的增加，集合发生扩容后部分元素的位置发生变化，元素的顺序就更加不能保证了。

  • 唯一：从HashMap的源码中我们可以了解到，HashMap是通过元素的hash值跟equals()方法来判断元素是否存在的

    并且从源码中可以得出一个结论：当hash值不同时，无需再比较equals()方法。

注意点：

  • 在使用HashSet的时候，一定要重写元素的hashCode()和equals()方法

    因为存在hash冲突的情况，hash值相同，但这两个对象不一定就相同，反之，hash值不同的两个对象，一定不相同；同时因为hash值相同，两个对象不一定相同，这个时候就需要用到equals()方法来进一步判断对象是否相同。

  • HashMap的扩容机制是当元素的数量到达扩容阈值（扩容因子*当前容量）时就会扩容一倍的容量，举例：
